# Saathi - Supabase Row Level Security (RLS) Policies

This directory contains the SQL script for managing Row Level Security (RLS) policies on the Supabase PostgreSQL database.

## RLS Management Script

The primary script is:
* [rls_policies.sql](file:///d:/Project%20Netra/Saathi/supabase/rls_policies.sql)

It secures the 7 core tables generated by the Django ORM:
1. `users_user`
2. `users_accessibilitypreferences`
3. `surveys_survey`
4. `surveys_question`
5. `responses_session`
6. `responses_sessionanswer`
7. `responses_auditlog`

---

## 1. When to Run the Script

> [!IMPORTANT]
> The RLS script must **only** be executed **after** Django migrations have run and successfully created the tables in the Supabase database. If you run the RLS script before migrations, the SQL engine will fail because the target tables do not yet exist.

### Step-by-Step Workflow:
1. Trigger backend deployment on Render (which runs Django release command `python manage.py migrate` automatically).
2. Verify in the Supabase Dashboard that the database tables are created.
3. Open the **SQL Editor** in the Supabase Dashboard.
4. Copy the entire contents of [rls_policies.sql](file:///d:/Project%20Netra/Saathi/supabase/rls_policies.sql) and paste it into the editor.
5. Click **Run**.

---

## 2. Verification Steps

To verify that Row Level Security has been correctly applied and is currently active:

### A. Verify RLS is Enabled
Run the following query in the Supabase SQL Editor:
```sql
SELECT tablename, rowsecurity 
FROM pg_tables 
WHERE schemaname = 'public' 
  AND tablename IN (
    'users_user', 
    'users_accessibilitypreferences', 
    'surveys_survey', 
    'surveys_question', 
    'responses_session', 
    'responses_sessionanswer', 
    'responses_auditlog'
  );
```
**Expected Output:** All 7 tables must return `rowsecurity = true`.

### B. Verify Policy Application
To check the active policies in the public schema, run:
```sql
SELECT schemaname, tablename, policyname, permissive, roles, cmd, qual 
FROM pg_policies 
WHERE schemaname = 'public';
```
**Expected Output:** You should see policy names like `Allow public read of active surveys` and `Allow session owner access` associated with their respective tables.

---

## 3. Rollback Steps

If you need to disable RLS policies (e.g., during database maintenance or troubleshooting):

1. Open the Supabase **SQL Editor**.
2. Run the following rollback block (which is also documented at the bottom of `rls_policies.sql`):
```sql
-- Disable Row Level Security (RLS) on all tables
ALTER TABLE IF EXISTS users_user DISABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS users_accessibilitypreferences DISABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS surveys_survey DISABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS surveys_question DISABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS responses_session DISABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS responses_sessionanswer DISABLE ROW LEVEL SECURITY;
ALTER TABLE IF EXISTS responses_auditlog DISABLE ROW LEVEL SECURITY;

-- Drop all defined policies
DROP POLICY IF EXISTS "Allow users to access own record" ON users_user;
DROP POLICY IF EXISTS "Allow users to access own preferences" ON users_accessibilitypreferences;
DROP POLICY IF EXISTS "Allow public read of active surveys" ON surveys_survey;
DROP POLICY IF EXISTS "Allow public read of questions" ON surveys_question;
DROP POLICY IF EXISTS "Allow session owner access" ON responses_session;
DROP POLICY IF EXISTS "Allow session owner to manage answers" ON responses_sessionanswer;
DROP POLICY IF EXISTS "Allow session owner to append logs" ON responses_auditlog;
```
3. Run the verification query from Section 2A; all tables should now return `rowsecurity = false`.
